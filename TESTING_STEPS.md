# 決賽 Prototype 測試步驟

最後更新：2026-06-29

本文件給評審、PM 與 QA 直接操作「從碳客變捷客」概念原型。原型不需要登入、不需要網路 API，也不會送出真實會員、票務、付款或區塊鏈資料。

## 測試入口

- 線上 prototype：<https://sofiayan0523.github.io/from-carbon-to-metro/>
- 本地 server：在 repo 根目錄執行 `python3 -m http.server 4173 --bind 127.0.0.1`，再開啟 <http://127.0.0.1:4173/>
- 離線包：`offline/from-carbon-to-metro-offline.zip`
- 建議瀏覽器：Chrome、Edge 或 Safari 的近代版本
- 狀態重置：重新整理頁面即可回到初始 demo state

注意：在開發分支尚未合併到 `main` 前，GitHub Pages production URL 仍可能顯示舊版。Phase 6 branch QA 請先用本地 server 或離線包驗證；送件前需合併到 `main` 並重新跑本文件的線上測試流程。

## 測試前提

- 主流程完全使用本機 mock data 與 deterministic JavaScript。
- AI 推薦不呼叫外部模型或商家 API；排序分數由本機商家資料與固定權重計算。
- 碳足跡公式固定為 `90 gCO2e/km x 通勤距離`。
- 綠點加成門檻固定為本月 5 kg -> `1.2x`、10 kg -> `1.5x`。
- Numbers proof 使用固定示範時間 `2026-06-29 09:41`，目的是讓離線 QA、截圖與現場 demo 可重播；正式產品會改用真實事件時間。
- Proof payload 的區間 bucket 採閉開區間概念：例如 50 km 會落入 `50-100 km`，15 趟會落入 `15-24 趟`。

## A. 線上測試流程

1. 開啟線上 prototype。
   - 預期看到頁首「從碳客變捷客」與四個上方切換按鈕：`首頁`、`AI 路線`、`碳足跡`、`可信證明`。
   - 手機框內預設位於 `首頁・今日綠點報告`。

2. 在首頁點選 `完成通勤`。
   - 預期本月減碳由 `8.2` 更新為 `8.9` kg。
   - 預期捷運點數由 `480` 更新為 `502`。
   - 預期累計趟數由 `18` 更新為 `19`。
   - 預期今日減碳顯示 `0.72 kg CO2`。
   - 預期第一個任務狀態變為 `已完成`。

3. 點選 `AI 路線`。
   - 預期地圖上有 6 個商家 pin，底部有推薦商家與 AI 分數理由。
   - 點選 `餐飲` 篩選後，預期推薦收斂到餐飲商家。
   - 點選 `點數最大` 或排序 `折抵`，預期推薦順序改變，且每張卡仍顯示原因。
   - 點選 `模擬累點`，預期點數依目前綠點倍率增加；點選 `用點兌換`，預期點數扣除。

4. 點選 `碳足跡`。
   - 預期看到 `90 gCO2e/km` 公式、5 kg / 10 kg 門檻與 14 天趨勢圖。
   - 切換 `短程轉乘`、`日常通勤`、`跨城挑戰`，預期公式中的距離與 CO2 值同步更新。
   - 選 `跨城挑戰` 後點選 `完成示範通勤`，預期本月減碳超過 10 kg，綠點倍率更新為 `1.5x`。

5. 點選 `可信證明`。
   - 若還沒完成通勤，點 `產生示範證明` 會顯示 `請先完成通勤`。
   - 完成通勤後，proof record 會顯示去識別 payload，事件 ID 預期為 `METRO-202606-0019`。
   - 未勾選隱私同意時，點 `產生示範證明` 會顯示 `等待隱私同意`，不會產生 hash。
   - 勾選 opt-in 後再點 `產生示範證明`，預期顯示 `NID-DEMO-51989C`、demo hash `0x51989C8D`、audit progress `100%`。
   - Proof 畫面不應顯示完整起訖站、商家、精確時間或精確通勤距離。

6. 回到首頁點選 `產生分享卡`。
   - 預期 modal 顯示本日減碳、本月累積、捷運點數與任務完成數。
   - 按 `Esc` 或右上角 `x` 可關閉。
   - 開啟 modal 時背景頁面不應繼續捲動。

## B. 離線測試流程

1. 將 `offline/from-carbon-to-metro-offline.zip` 複製到沒有網路的電腦。

2. 解壓縮後進入 `from-carbon-to-metro/` 資料夾。

3. 優先用本地 server 開啟：

   ```bash
   python3 -m http.server 4173 --bind 127.0.0.1
   ```

   開啟：

   ```text
   http://127.0.0.1:4173/
   ```

4. 如果現場電腦沒有 Python，直接用瀏覽器開啟解壓後的 `index.html` 作為 fallback。
   - Demo 邏輯仍可執行。
   - 若特定瀏覽器限制 `file://` 字型載入，畫面會回退到系統中文字型，不影響主流程判讀。

5. 依照「A. 線上測試流程」第 2 到第 6 步重跑一次。
   - 預期結果應與線上流程一致。
   - 原型不應出現等待 API、登入、付款、外部圖片、區塊鏈 RPC 或 Capture / Numbers live API 的阻塞狀態。

## C. 桌面與手機版檢查

1. 桌面寬度測試。
   - 使用一般筆電或桌機瀏覽器開啟。
   - 四個區塊內容可捲動，底部導覽列不應遮住主要按鈕。
   - 分享卡 modal 開啟後不應溢出畫面。

2. 手機寬度測試。
   - 使用瀏覽器 responsive mode，建議寬度 390 px 或 420 px。
   - 四個上方切換按鈕、手機框、底部導覽列不應水平溢出。
   - Proof record 的長 payload 可換行，不應撐破卡片。

## D. 素材與隱私檢查

- Repo 內唯一打包字型為 `assets/fonts/NotoSansCJKtc-Regular.otf`，授權文字在 `assets/fonts/OFL.txt`。
- 原型沒有使用官方捷運 Logo、未授權圖片、音樂、地圖底圖或第三方品牌素材。
- 商家名稱只作為文字 mock 情境，不含商標圖、廣告圖或交易資料。
- Proof payload 只保留月份、減碳值、趟數區間、距離區間、任務類型、倍率、同意狀態與 demo hash。
- 主流程沒有 secrets、tokens、API key、真實會員資料、真實票務紀錄或付款資料。

## E. 通過標準

- 線上 URL 可完成首頁 -> AI 路線 -> 碳足跡 -> 可信證明 -> 分享卡主流程。
- 離線包在無網路環境可完成同一主流程。
- `產生示範證明` 在 opt-in 後可重現 `NID-DEMO-51989C` 與 `0x51989C8D`。
- 任何 live integration 文案都只能是加分展示，不得阻塞主 demo。
- 桌面與手機寬度下沒有主要文字、按鈕、卡片或底部導覽列互相覆蓋。
